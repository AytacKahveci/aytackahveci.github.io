Architecture:

Computer Vision and High-Level Control:

    • Custom image processing techniques and filters for object localization.
    • A Kalman filter-based multi-frame object tracker for temporal consistency and noise rejection.
  • High-level robot control logic, including target pose computation and movement sequencing, was implemented in Python and integrated with the ROS ecosystem.

Robot Integration Layer:

  • Implemented a custom ROS-compatible KUKA robot driver in C++, acting as a middleware between ROS nodes and the KUKA internal robot controller via real-time TCP/IP or FRI (Fast Robot Interface).

Visual Servoing Loop:

  • Real-time camera input processed to detect the object pose in image space.

  • Robot motion goals were calculated dynamically and sent to the KUKA controller via ROS action interfaces.

Frontend Interface:

  • Built a Qt-based GUI for visualizing:
    • Live camera feed with detection/tracking overlays.
    • Real-time joint states and robot pose.

System Integration & Calibration:

  • Calibrated the camera-to-robot frame using hand-eye calibration techniques.
  • Verified visual servoing accuracy using test patterns and repeated tasks under varying object positions and lighting.

Challenges & Solutions:

📡 Real-Time Object Tracking and Actuation:

Challenge: Vision processing needed to keep up with real-time robot feedback to ensure stable servoing.

Solution: Optimized the Kalman-based tracking loop and ensured synchronization with ROS time; implemented rate-limited command publishers to avoid command saturation.

🤝 Bridging KUKA Internal Driver with ROS:

Challenge: The proprietary KUKA driver did not natively support ROS.

🖥 Usability and Operator Control:

Challenge: Need for monitoring and debugging during development and operation.

Solution: Designed an intuitive Qt GUI that allowed toggling between manual and auto modes, displaying real-time feedback, and logging debug info for replay.

Autonomous Mobile Robot Low-Level Controller

I developed a low-level motion controller for an autonomous mobile robot to manage precise motor control, odometry feedback, and real-time command execution.

C++ Python ROS TCP/IP Linux

I developed a low-level motion controller for an autonomous mobile robot to manage precise motor control, odometry feedback, and real-time command execution. The controller acted as the bridge between high-level navigation systems and physical motor drivers, ensuring safe and accurate motion execution.

Architecture:

Platform:

  • Developed entirely in C++ under a Linux environment with strict real-time performance considerations.
  • The controller was implemented as a ROS (Robot Operating System) node, enabling modular integration with upstream navigation and localization systems.

Communication:

  • TCP/IP socket communication was used for interfacing with the robot’s embedded motor driver board.
  • The protocol supported high-frequency bidirectional messaging for encoder feedback, motion commands, and fault reporting.

Control Loop:

  • Implemented a real-time control loop (~100Hz) that:
    • Received target velocities via ROS cmd_vel topics
    • Translated velocity commands into wheel-level PWM or encoder commands
    • Parsed encoder data to compute odometry and published it to ROS as odom
  • Implemented PID control logic for smooth acceleration, deceleration, and heading correction.

Diagnostics and Safety:

  • Included watchdog timers and command timeouts to prevent runaway behavior.
  • Implemented overcurrent and overtemperature detection based on motor feedback.
  • Provided emergency stop and fault recovery service endpoints through ROS.

Integration and Tools:

  • Integrated with RViz for real-time visualization of odometry and motion vectors.
  • Custom Python-based test harness was developed for stress testing the controller over TCP/IP.
  • Unit-tested with GTest, and deployment was done using CMake and ROS launch files.

Challenges & Solutions:

⚡ Low Latency Communication:

Challenge: Communication with the motor driver had to be deterministic and fast to maintain stable control.

Solution: Designed a lightweight binary protocol over TCP, minimized serialization overhead, and used non-blocking sockets with epoll for efficient I/O under Linux.

📏 Accurate Odometry and Drift Reduction:

Challenge: Wheel slippage and encoder noise affected odometry precision.

Solution: Implemented filtering and rolling average smoothing for encoder deltas; allowed real-time calibration through ROS services for wheelbase and ticks-per-meter.

🔁 Synchronization with ROS Time and Frames:

Challenge: Maintaining timestamp alignment with other subsystems (e.g., localization).

Solution: Used ros::Time::now() consistently for odometry messages and adhered to the TF transform tree (base_link → odom) for proper integration.

🛑 Safety and Fault Handling:

Challenge: Ensuring that faults (e.g., lost packets, stalled motors) didn’t propagate upstream.

Solution: Added a state machine inside the controller that reported OK, FAULT, and EMERGENCY_STOP states, each with specific recovery behavior and ROS status topics.

Autonomous Mobile Robot Object Pose Estimation

I developed a robust object pose estimation pipeline for an autonomous mobile robot, enabling it to perceive and track objects in 3D space using real-time point cloud data.

C++ Python PCL ROS RViz Gazebo Linux Docker

I developed a robust object pose estimation pipeline for an autonomous mobile robot, enabling it to perceive and track objects in 3D space using real-time point cloud data. The system enhanced raw sensor input, filtered noise, and estimated object poses with temporal consistency, forming a crucial part of the robot’s perception and manipulation stack.

Architecture:

Sensor Input and Processing:

  • Input was acquired from a 3D LiDAR sensor, delivered as Point Cloud Data (PCD).
  • Implemented custom point cloud preprocessing filters in C++ using the Point Cloud Library (PCL):
    • Voxel Grid Filter to downsample data
    • Statistical Outlier Removal to eliminate noisy measurements
    • PassThrough Filter to crop to the region of interest

Object Detection and Pose Estimation:

  • Extracted object clusters using Euclidean Cluster Extraction.
  • Calculated bounding boxes, centroids, and principal axes for detected objects.
  • Implemented a Kalman Filter in C++ and Python to fuse current and historical observations for smoother and more reliable pose estimation.
    • Filter tracked 6-DoF pose: (x, y, z, roll, pitch, yaw)
    • Managed uncertainty using measurement and process noise covariance tuning

System Integration:

  • Pose estimation pipeline was integrated into a ROS (Robot Operating System) node, with real-time publishing of object poses.
  • Data was synchronized and visualized in RViz, with support for replay via ROS bag files for debugging and testing.

Post-processing and Smoothing:

  • Applied temporal smoothing and outlier rejection across frames using a pose consistency checker to handle dynamic environments and occlusion cases.

Challenges & Solutions:

📉 Noisy Point Cloud Data:

Challenge: Raw point cloud data had high noise, reflections, and artifacts.

Solution: Designed a modular filter stack (voxel grid + outlier removal + ROI cropping) that could be tuned per environment and object size.

⏱️ Real-Time Constraints:

Challenge: Processing high-density point clouds without lagging downstream motion planning modules.

Solution: Used multi-threading and batch downsampling techniques; computation-heavy stages (clustering, filtering) were moved to C++ with ROS async callbacks.

🔄 Pose Drift and Identity Confusion:

Challenge: In dynamic environments, the object ID and pose tracking could drift or jump.

Solution: Kalman Filter with identity association logic based on object size, motion pattern, and spatial proximity. Used Mahalanobis distance for matching predictions to measurements.

📦 Sensor Fusion Support (Future-Ready):

Although not implemented in full, the architecture was designed to allow future fusion of vision-based pose detection with point cloud estimations using extended Kalman filters.

I designed and implemented a high-precision docking system for an autonomous mobile robot to enable consistent and repeatable alignment with charging stations or loading platforms.

C++ Python ROS Gazebo Kalman Filter Path Planning Trajectory Planning Closed Loop Control Linux

I designed and implemented a high-precision docking system for an autonomous mobile robot to enable consistent and repeatable alignment with charging stations or loading platforms. The system leveraged feedback control and real-time pose updates to achieve centimeter-level accuracy under varying environmental conditions.

Architecture:

Control Algorithm:

  • Implemented a PID-based motion controller in C++, designed to minimize the robot's positional and angular error relative to the docking target.
  • Controller supported both linear and angular velocity adjustments, with dynamic PID parameter tuning for smooth deceleration as the robot approached the target.

Target Pose Acquisition:

  • Docking targets were defined as fixed Pose2D or Pose3D coordinates relative to the robot’s global map.
  • Final docking pose was broadcast as a ROS static transform or dynamically estimated from markers.

Integration with Navigation Stack:

  • Docking procedure was integrated as a stateful ROS node that:
    • Took control from the high-level planner upon reaching the docking zone
    • Switched from global navigation to fine-grained pose alignment
    • Monitored docking success using position thresholds and timeout logic

Feedback and Safety:

  • Position feedback was obtained from odometry and localization (e.g., AMCL or EKF fused sensors).
  • Safety mechanisms were implemented:
    • Timeout and maximum attempt limits
    • Emergency stop triggers on error spikes or sensor failure

Simulation & Testing:

  • Extensive validation was conducted in Gazebo simulation and real robot deployments.
  • ROS bag data was recorded and replayed to test docking accuracy, repeatability, and recovery from partial failures.

Challenges & Solutions:

🎯 Accuracy and Stability Near Target:

Challenge: Robot tended to overshoot or oscillate as it approached the docking point.

Solution: Tuned PID controller with variable gain scaling near the target. Introduced a velocity clamp and arrival window threshold to define "docked" status with precision.

🔁 Transition from Global to Local Control:

Challenge: Switching from global navigation to docking logic required a clean handover without trajectory discontinuity.

Solution: Created a controller activation threshold that smoothly transitioned control authority based on distance to docking goal.

⏱️ Timing and Feedback Latency:

Challenge: Delays in sensor feedback could cause delayed adjustments and degrade docking performance.

Solution: Used a high-frequency control loop (~50–100Hz) and synchronized pose updates with ROS TF and odometry timestamps.

⚠️ Safety and Fault Handling:

Challenge: Misalignment or false success detection due to sensor drift or mechanical noise.

Solution: Implemented a multi-condition success validator, including position error, angle error, and motion stability check over a short window.

Multi Robot Fleet Traffic Management

Python MQTT ZeroMQ MySQL Redis React Flask Kubernetes

I designed and implemented a traffic management system for coordinating autonomous mobile robots (AMRs) in a shared environment, such as a smart warehouse or industrial floor. The objective was to ensure collision-free navigation, optimal path allocation, and real-time conflict resolution across multiple robots operating simultaneously.

Architecture:

Communication Layer:

  • MQTT was used for real-time bidirectional communication between robots and the central traffic controller.
  • ZeroMQ was used for high-speed inter-service communication for path planning, conflict resolution, and logging modules.

Traffic Coordination Logic:

  • A centralized Python-based controller using multithreading and async IO handled:
    • Robot position updates
    • Traffic zone reservations (virtual lanes, intersections)
    • Deadlock prevention using graph-based dependency resolution
  • A* and Dijkstra's algorithms were used for dynamic route planning based on live robot positions and blocked paths.

Storage:

  • MySQL stored robot telemetry, traffic zone states, and event logs.
  • Redis was used as an in-memory store for fast access to current robot positions and traffic locks.

Frontend:

  • A React-based live dashboard visualized:
    • Real-time robot positions on a map
    • Traffic heatmaps
    • Active conflicts and resolutions
  • Backend exposed RESTful APIs via Flask

CI/CD and Deployment:

  • GitHub Actions for CI, automated testing, and container builds
  • Docker + Kubernetes for orchestrating services like route planner, conflict resolver, and monitor
  • Prometheus + Grafana for monitoring robot traffic KPIs (average speed, congestion zones, etc.)

Challenges & Solutions:

Collision & Deadlock Prevention:

Challenge: Prevent robots from entering the same traffic zone simultaneously.

Solution: Implemented a zone-based reservation system with lease timeouts and a priority queue for contested zones. Robots reserve zones in advance before movement and renew as they proceed.

Real-time Responsiveness:

Challenge: Position updates and route changes needed to be processed under 200ms.

Solution: Used async MQTT clients, batching updates, and event-driven scheduling to reduce processing overhead.

Debugging and Monitoring:

Challenge: Diagnosing traffic jams or missed reservations required insight into real-time decisions.

Solution: Built a trace logger for each robot’s traffic decisions and visualized them on the dashboard. Grafana alerts were configured for high contention zones or increased deadlocks.

I contributed a high-performance map provisioning module to parse, structure, and deliver road topology data to the autonomous vehicle control stack.

C++ Python Adaptive AUTOSAR NDS SQlite Docker QEMU QNX OS

I developed a high-performance map provisioning module to parse, structure, and deliver road topology data to the autonomous vehicle control stack. The system processed NDS (Navigation Data Standard) map data received from a remote service, constructed detailed road network models (e.g., links, lanes, and intersections), and supplied them in real time to downstream motion planning and localization components.

Architecture:

Platform: Implemented in C++ under QNX OS with strict real-time constraints and safety requirements.

Integrated into the Adaptive AUTOSAR framework as an ARA::COM service, enabling modular, reusable, and safety-compliant deployment.

Data Ingestion: NDS map data was received over a vehicle communication bus SOME/IP from an external navigation provider.

Map tiles and segment data were parsed incrementally to support streaming behavior and low memory footprint.

Map Parsing and Structuring: Implemented a custom map parser that decoded NDS binary payloads into structured internal representations:

  • Links: Represented road segments, with attributes such as geometry, speed limits, and directionality
  • Lanes: Included lane markings, connectivity, type (e.g., drivable, restricted), and hierarchy
  • Intersections & Lane Connections: Constructed topological relationships used by behavior planning modules

Maintained a spatial index for fast querying of map elements based on vehicle position.

Integration with Control Stack: Exposed structured map data as Adaptive AUTOSAR service interfaces to consumers like:

  • Trajectory planners
  • Localization (e.g., map-matching)
  • Traffic rule engines

Implemented change notifications and region-based data provisioning to reduce bandwidth and CPU usage.

Challenges & Solutions:

🧱 Parsing Complex Binary Formats:

Challenge: NDS data format is dense and highly nested, making parsing prone to errors and performance issues.

Solution: Designed a layered decoder architecture with robust error handling and buffer reuse to handle partial and malformed data gracefully.

⏱️ Real-Time Performance on QNX:

Challenge: Real-time deadlines under QNX with limited system resources.

Solution: Used lock-free data structures and pre-allocated memory pools to ensure deterministic behavior. Parsed map segments asynchronously and published results via ARA::COM with QoS settings.

📍 Dynamic Region Loading:

Challenge: Vehicle moved across regions, requiring dynamic map tile switching.

Solution: Implemented a region manager that requested and loaded only nearby map segments based on vehicle GPS, supporting seamless handover between tiles.

🔐 Safety and Certification Constraints:

Challenge: Compliance with automotive safety standards under Adaptive AUTOSAR.

Solution: Followed MISRA-C++ guidelines, ensured service availability states matched the vehicle mode, and validated each map element with checksum and schema versioning.

I worked on a Certificate Management and Map Validation system, where the goal was to ensure the authenticity and integrity of signed map data received from external sources. The core of the project was to securely manage digital certificates, validate signed maps using these certificates, and build an architecture that could handle high-throughput map verification in real-time scenarios.

Architecture Overview:

  • Certificate Handling: Certificates were securely retrieved from a remote certificate authority over HTTPS.
  • Map Validation Service: A C++ based validation service that used cryptographic libraries (e.g., OpenSSL) to verify signatures on incoming map files.
  • Storage: Certificates and validation logs were stored in a secure PostgreSQL database.
  • Frontend: Admin dashboard for monitoring certificate status and validation outcomes.
  • Deployment: Containerized with Docker, orchestrated via Kubernetes, and CI/CD pipelines handled via GitHub Actions.

Key Challenges & Solutions:

🔐 Security & Trust Management:

Challenge: One of the biggest challenges was maintaining trust and preventing misuse of certificates.

Solution: We implemented strict validation chains and expiration checks, and ensured that all certificate downloads and storage were encrypted at rest and in transit.

⚙️ Real-Time Map Validation:

Challenge: Another challenge was verifying large, signed maps in real-time without blocking the main processing pipeline.

Solution: To solve this, we designed an asynchronous validation worker system using message queues (RabbitMQ), which allowed map validation to run in parallel and report back results without latency overhead.

Impact:

This architecture allowed the system to securely validate maps at scale, maintain a chain of trust, and provide real-time feedback on map authenticity. It also laid the groundwork for future integration with other secure vehicle or navigation systems where verified map data is critical.

Event Driven Framework development in Turbine Software Control

I contributed to an event-driven framework where multiple system components communicated by sending and responding to events.

C++ Python Boost Message Queue Kubernetes Linux QNX OS

I contributed to an event-driven framework where multiple system components communicated by sending and responding to events. The main goal of the project was to decouple component interactions and ensure real-time responsiveness through a scalable and maintainable architecture.

Architecture Overview:

  • Event Bus: Boost Message Queue to handle asynchronous message passing between components.
  • Producers: Individual components publish events to the event bus when their internal state changes.
  • Consumers: Other components subscribe to relevant event types and update themselves accordingly.
  • Event Manager: A centralized C++ based framework responsible for routing events and managing subscription metadata.
  • Deployment: The system was containerized and deployed using Kubernetes, with horizontal scaling for high-load event routing.

Key Challenges & Solutions:

🔄 Loose Coupling and Dynamic Subscription Management:

Challenge: One major challenge was designing a flexible and dynamic subscription model that didn’t tightly couple components.

Solution: We solved this by implementing a central event registry where components could register their interest in specific event types at runtime.

⚙️ Event Ordering and Delivery Guarantees:

Challenge: We needed to ensure that event consumers processed messages in the correct order and without loss, especially for critical system events.

Solution: To address this, we used Boost Message Queue with persistent queues and message acknowledgements, and enforced ordering using per-component routing keys.

📈 Scalability and Performance:

Challenge: As the number of components and event types grew, routing efficiency became critical.

Solution: We scaled the event dispatcher horizontally using Kubernetes, and also introduced event batching and compression for high-frequency events to reduce latency.

Impact:

This architecture enabled highly decoupled, scalable, and reactive systems. Components could be added or removed with minimal impact, making it ideal for modular system design. It also significantly improved real-time responsiveness and fault tolerance, as failed consumers could recover from missed events via queue replay.
